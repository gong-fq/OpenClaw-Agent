大语言模型LLM
▼

大语言模型（Large Language Model，LLM）是基于 Transformer 架构、在海量文本数据上预训练的深度神经网络。其核心机制是自回归预测——给定上下文，预测下一个 token 的概率分布。

这一机制赋予了 LLM 出色的语言理解与生成能力，但也决定了其本质：LLM 是概率语言预测器，而非逻辑推理引擎或精确计算工具。理解这一点，是本课程的起点。正因 LLM 无法保证结构一致性，才需要 Agent 框架将其与 Tool、Memory、Controller 结合，补足其固有缺陷。

  • Transformer + 自注意力机制是基础架构，GPT 系列为典型代表
  • 训练范式：预训练（Pre-training）+ 指令微调（SFT）+ RLHF 对齐
  • 涌现能力（Emergent Abilities）：规模超过阈值后出现的推理跃迁
  • 上下文窗口（Context Window）是短期记忆的边界，超出则遗忘
  • 幻觉（Hallucination）：以高置信度输出错误事实，是 Agent 框架存在的核心动因
  • DeepSeek、ChatGLM、Qwen 是目前主流的国产开源 LLM

智能体Agent
▼

AI Agent（智能体）是一种以 LLM 为"大脑"，能够自主感知环境、制定计划、调用工具并持续执行直至完成目标的程序系统。与单轮问答不同，Agent 实现了从"被动应答"到"主动做事"的根本转变。

核心公式：Agent = LLM（大脑）+ Planning（规划）+ Tool Use（工具）+ Memory（记忆）

在 OpenClaw 框架中，Agent 的行为由 Controller 统筹调度，Planner 负责推理分解，Executor 调用 Tool，Memory 管理状态信息——五模块协同，构成完整的自主执行系统。

  • 核心特征：自主性（Autonomy）、目标导向、多步执行、环境交互
  • ReAct 范式：Reasoning（推理）+ Acting（行动）交错进行，是主流 Agent 推理框架
  • 吴恩达 4 大设计模式：Reflection · Tool Use · Planning · Multi-Agent 协作
  • Agent ≠ RPA：Agent 具备语义理解和动态推理，而非固定流程脚本
  • 幻觉控制、错误重试、循环终止是工程实现的核心挑战
  • 2025 年被称为"Agent 元年"，多智能体协作（Multi-Agent）是前沿方向

控制器Controller
▼

Controller（控制器）是 Agent 框架中最容易被忽视、却最关键的模块——它是整个执行循环的"大脑干"，负责协调所有其他模块。Controller 决定：何时启动新推理步骤、何时调用工具、何时终止循环，以及发生错误时如何重试或降级处理。

没有 Controller，LLM 可能陷入无限循环、资源耗尽，或在错误路径上无法自救。这是本课程的差异化优势讲次：大多数 Agent 教程只讲 Planner 和 Tool，忽视了 Controller 才是工程稳定性的核心保障。

  • 执行循环：step → observe → plan → act → repeat（直到终止条件触发）
  • 终止条件：最大步数限制 · 目标完成检测 · 异常捕获
  • 重试机制（Retry）：对 LLM 输出格式错误、Tool 调用失败的容错处理
  • 无限循环防护：防止 Agent 在无解任务上永久运行，消耗 API Token
  • 类比：操作系统的进程调度器；人脑前额叶皮层的执行功能
  • LangGraph 是目前最主流的基于状态机的 Controller 实现框架

规划器Planner
▼

Planner（规划器）是 Agent 中负责"思考"的模块，其底层就是 LLM 本身。它接收当前任务和上下文，输出下一步行动计划。主流方法包括思维链（CoT）、ReAct（推理+行动交错）和思维树（ToT）。

课程核心洞见：Planner 是语义预测系统，不是精确计算系统。它擅长语义分解与自然语言推理，但面对精确数学、空间几何、结构约束问题时往往失败——这正是 Tool 不可缺少的原因，也是第 3、4 讲实验的核心内容。

  • Chain-of-Thought（CoT）：引导 LLM 输出中间推理步骤，显著提升复杂任务表现
  • Zero-shot CoT：追加"让我们一步步思考"即可激活推理能力，无需样本
  • ReAct：Thought → Action → Observation 循环，将推理与环境交互融合
  • Tree-of-Thoughts（ToT）：多路径规划，通过评估选择最优推理树
  • 任务分解（Task Decomposition）：将复杂目标拆解为可执行子任务序列
  • 规划失败点：空间推理、精确计数、多步约束——详见第 6 讲语义空间 vs 物理空间

记忆系统Memory
▼

Memory（记忆）是 Agent 的信息存储与检索层。LLM 本身没有持久记忆——每次调用独立、上下文有限。Memory 模块弥补了这一缺陷，使 Agent 能够跨步骤、跨会话地保持状态一致性。

OpenClaw 课程将 Memory 分为两类：JSON 状态记忆（结构化、精确，适合任务状态跟踪）和向量语义记忆（通过 RAG 检索相关上下文，适合知识库问答）。两者有本质区别，不可混淆——这是第 5 讲的核心实验内容。

  • 短期记忆（Short-term）：上下文窗口内的对话历史，会话结束即消失
  • 长期记忆（Long-term）：向量数据库 + 语义检索，支持跨会话知识持久化
  • 情景记忆（Episodic）：存储 Agent 历史行为序列，支持反思与改进
  • 程序性记忆（Procedural）：System Prompt、工具定义、行为规则
  • RAG（检索增强生成）：将外部知识库与 LLM 生成能力结合的主流实践
  • 向量嵌入（Embedding）：文本映射为高维向量，支持语义相似度检索（FAISS/Chroma）

工具系统Tool
▼

Tool（工具）是 Agent 通向外部结构系统的"手"。当 Planner（LLM）面对精确计算、数据库查询、代码执行、API 调用等结构性任务时，必须将控制权交给对应工具——这是本课程的核心工程洞见。

工具系统的现代标准是 Function Calling：LLM 以结构化 JSON 格式输出工具名称和参数，由 Controller 调度执行，结果返回 LLM 继续推理。2024 年 Anthropic 发布的 MCP（Model Context Protocol）进一步标准化了 Agent 与工具之间的接口协议，是当前最重要的行业标准之一。

  • Python Tool：代码执行能力，解决精确计算、数据处理、算法问题
  • API Tool：调用外部服务，如天气、搜索引擎、数据库、支付接口
  • Function Calling：LLM 输出结构化调用描述，由框架执行并返回结果
  • MCP（Model Context Protocol）：Anthropic 2024 年提出的 Agent-Tool 标准接口协议
  • 工具描述（Tool Description）的质量直接决定 LLM 能否正确选择和调用工具
  • 核心原则：结构计算引擎做精确的，LLM 做语义模糊的，各司其职
